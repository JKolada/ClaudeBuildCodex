Dekalog — rdzeń doktryny

  1. I
    Dokumentuj dla agenta, nie dla archiwum.
    Każdy katalog ma AI_README; aktualizuj go przed commitem, nie po.
  2. II
    Szukaj w historii, zanim napiszesz linijkę.
    git log -S / --grep / blame + AI_README. Ktoś już to rozwiązał w tym repo.
  3. III
    Weryfikuj, nie deklaruj.
    Nie „działa" — dowód: smoke test, kod HTTP, liczby, screenshot.
  4. IV
    Dry-run jest domyślny; --execute świadomy.
    Skrypt mutujący najpierw pokazuje plan. Mutacja po przeczytaniu.
  5. V
    Backup to mechanizm rollbacku, nie ostrożność.
    Migracje forward-only; snapshot przed każdą zmianą na prodzie.
  6. VI
    Prod jest święty — nie dotykasz bez „wdrażaj".
    Commit i push to nie deploy. Nieodwracalne potwierdzaj zawsze.
  7. VII
    Dane użytkownika są nienaruszalne.
    Wylicz każdą tabelę FK; mapuj po slug, nie ID; źródło = żywy prod; zweryfikuj integralność.
  8. VIII
    Taguj każdy deploy i pisz, co user zyskał.
    Annotated tag = „co jest live". Changelog prostym językiem, bez żargonu.
  9. IX
    Mały, spójny commit; jeden temat.
    Rozdzielaj niezwiązane zmiany. Czysty git status; wykrywaj śmieci i sieroty wcześnie.
  10. X
    Plan → iteruj → review.
    Najpierw szkic (3, nie 100). Liczby weryfikuj. UX dowieź z ciepłem.

Złota zasada altytudy

Wolno tam, gdzie błąd jest drogi (prod, dane userów, schemat). Szybko tam, gdzie jest tani (lokalny eksperyment, szkic UI, dry-run). Tempo dobieraj do kosztu pomyłki, nie do niecierpliwości.

Siedem grzechów głównych

1. Nadęty optymizm w raporcie („wszystko działa" bez dowodu).
2. Mutacja bez planu (od razu --execute).
3. Auto-deploy („skoro gotowe, to wrzucam").
4. Swap „prócz kont" jako jedna tabela users.
5. Commit-śmietnik (niezwiązane zmiany + artefakty).
6. Skalowanie przed review (500 sztuk, nim user zobaczył 3).
7. Dokumentacja „później".
